KEY FINDINGS
================================================================================

1. EVENTBUS ARCHITECTURE (src/amoskys/eventbus/server.py)
   ✓ IMPLEMENTED:
     - gRPC server with mTLS (client certs required)
     - Size validation (max 131KB by default)
     - Idempotency deduplication (300 second TTL, 50K entry cache)
     - Backpressure via inflight request tracking
     - Ed25519 signature verification (optional, REQUIRE_SIGNATURES=false default)
     - Prometheus metrics (7 metrics: publish_total, invalid_total, latency_ms, inflight, retry_total, dedup_hits, wal_failures)
     - Graceful shutdown (SIGHUP/SIGTERM with 10s grace period)
     - Health check endpoint (GET /healthz on port 8080)

   ✗ NOT IMPLEMENTED:
     - Authorization via CN checking (trust map loaded but never enforced)
     - Streaming Subscribe RPC (returns UNIMPLEMENTED)
     - Full readiness checks on health endpoint

   ⚠ ISSUES:
     - Signatures required by default: FALSE (backward compat risk)
     - Trust map never consulted (any valid client cert accepted)

2. WAL LAYER ARCHITECTURE (src/amoskys/agents/flowagent/wal_sqlite.py)
   ✓ FULLY IMPLEMENTED:
     - SQLite WAL mode (synchronous=FULL for crash safety)
     - BLAKE2b checksums (32-byte digest, detects corruption)
     - Hash chain (BLAKE2b(env || prev_sig) for tamper detection)
     - Genesis state (32 zero bytes for chain start)
     - Idempotency (UNIQUE constraint on idem key)
     - Backpressure (tail-drop, auto-deletes oldest when >200MB)
     - Vacuum triggers (reclaims space when >30% deletions)
     - Thread-safe operations (RWLock on critical sections)

   SEMANTICS:
     - Write: checksum + chain sig computed and verified before INSERT
     - Read: Batch processing with checksum & chain verification
     - Quarantine: Failed events sent to wal_dead_letter table
     - Thread safety: Lock held during prev_sig read, chain compute, insert

       peripheral_events, dns_events, audit_events, persistence_events, fim_events,
       incidents, alert_rules, metrics_timeseries, wal_archive, wal_dead_letter
     - Indexes on: timestamp_ns DESC, device_id, risk_score DESC, type/category
     - Foreign key refs: None (denormalized for perf)

   ✓ MIGRATIONS:

     - 002: Add GeoIP/ASN columns (geo_*, asn_*)

     - Auto-applied on TelemetryStore init (lines 492-503)

   ROUTING:
     - process_events: From ProcessEvent or proc-agent security events
     - flow_events: From FlowEvent or flow-agent security events
     - security_events: From SecurityEvent sub-messages
     - dns_events: From dns-agent security events
     - audit_events: From kernel_audit security events
     - persistence_events: From persistence-agent security events
     - fim_events: From fim-agent security events
     - peripheral_events: From peripheral_agent STATUS events

4. ENRICHMENT PIPELINE (src/amoskys/enrichment/)
   ✓ THREE STAGES (ordered):
     1. GeoIP: MaxMind GeoLite2-City → geo_src_*, geo_dst_* fields
        - Caches 10K IPs (LRU)
        - Skips private IPs (127.*, 10.*, 192.168.*, 172.16.*, etc.)
        - Returns: country, city, latitude, longitude, continent, timezone

     2. ASN: MaxMind GeoLite2-ASN → asn_src_*, asn_dst_* fields
        - Caches 10K IPs (LRU)
        - Classifies networks: hosting, tor, vpn, education, government, residential, corporate
        - Returns: asn_number, asn_org, network_type, is_hosting/is_tor/is_vpn flags

     3. ThreatIntel: Local SQLite → threat_intel_match, threat_source, threat_severity
        - Caches 10K checks (LRU with 3600s TTL)
        - Checks: src_ip, dst_ip, source_ip, domain, hostname, file_hash, sha256, new_hash
        - Returns: matched (bool), severity (critical/high/medium/low), source

   DETERMINISM:
     - Same input → same output (unless DB contents change)
     - Failures graceful: if stage unavailable, continues with others
     - Status: "raw" (no stages), "partial" (some failed), "enriched" (all succeeded)

5. END-TO-END FLOW
   T0: Event created by agent
   T1: Publish(Envelope) RPC to EventBus
       - Validate size, sig, dedup, inflight
       - Extract FlowEvent
       - Write to WAL with checksums & chain sig
       - Return OK("accepted") to client
   T2: Event in WAL (durable, encrypted by SQLite)
   T3: WAL Processor running (separate process)
       - Read batch from WAL
       - Verify BLAKE2b checksums
       - Verify hash chain integrity
       - Parse protobuf
       - Enrich via pipeline
       - Insert into appropriate domain table
       - DELETE from WAL (commit)
   T4: Event in TelemetryStore
       - Queryable from dashboard
       - Retained per retention policy

================================================================================
CRITICAL ISSUES & GAPS
================================================================================

SECURITY:
1. Signatures optional by default (REQUIRE_SIGNATURES=false)
   → Fix: Set to true in production config

2. Trust map loaded but never checked
   → Impact: Any client with valid cert accepted regardless of CN
   → Fix: Call _peer_cn_from_context() and validate against TRUST dict

3. Overload mode can be toggled externally
   → Severity: Medium (availability attack)
   → Mitigation: Already logged with source tracking

RELIABILITY:
4. WAL Processor is separate process (not integrated)
   → Impact: Events accumulate in WAL if processor crashes
   → Fix: Monitor WAL row count and processor health

5. Dedup cache expires after 300 seconds
   → Issue: Clients retrying after TTL create duplicates
   → Fix: Use persistent dedup table, not memory cache

6. Events ACK'd before storage persistence
   → Semantics: "ACK after WAL" but not "ACK after store"
   → Impact: Dashboard may not show event immediately

7. Quarantine without replay
   → Dead letter queue is permanent
   → Impact: Corrupted events never recovered
   → Fix: Add replay mechanism

PERFORMANCE:
8. Signature verification on every request
   → With 10k RPS, adds ~100-200ms latency
   → Fix: Batch or cache verification

9. No request batching in WAL write
   → Each event: individual SQLite INSERT + COMMIT
   → Fix: Batch 100 events per transaction

OBSERVABILITY:
10. No metrics for hash chain breaks
    → Add: wal_chain_breaks_total counter

11. No tracing for single-event lifecycle
    → Add: Idempotency key in all logs

TESTING RECOMMENDATIONS
================================================================================

FUNCTIONAL:
[ ] Publish with valid envelope → stored in WAL → processed → in TelemetryStore
[ ] Publish duplicate (same idempotency_key) → OK("duplicate") returned
[ ] Publish oversized envelope → INVALID response
[ ] Publish without signature (REQUIRE_SIGNATURES=false) → OK
[ ] Publish with invalid signature (REQUIRE_SIGNATURES=true) → INVALID
[ ] Overload mode enabled → all requests return RETRY

RELIABILITY:
[ ] WAL Processor crash → events queue in WAL, resume when processor restarts
[ ] EventBus crash → events durable in WAL, resume processing when server restarts
[ ] Database corruption → BLAKE2b detects, sends to dead_letter table
[ ] Chain tampering → Hash chain verification catches modification

PERFORMANCE:
[ ] 1k RPS sustained → measure latency, inflight requests
[ ] 10k RPS burst → measure backpressure, RETRY rate
[ ] 200MB WAL → verify vacuum triggers and disk reclamation

SECURITY:
[ ] Client cert validation → reject invalid certs
[ ] mTLS enforcement → verify EVENTBUS_REQUIRE_CLIENT_AUTH=true
[ ] CN authorization → (once implemented) verify only trusted agents accepted
